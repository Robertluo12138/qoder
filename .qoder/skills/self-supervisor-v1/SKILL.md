# self-supervisor-v1

A Qoder CLI-compatible self-supervisor. It takes a natural-language
request and drives it through the full delivery lifecycle with
**deterministic audit checks**, so acceptance never relies solely on
model judgment.

## When to use this skill

Invoke when:

- the user says "build X", "add feature Y", "refactor Z", or otherwise
  describes a development task in natural language;
- the user asks for an end-to-end run that covers planning,
  implementation, testing, review, and verification;
- the user wants a sealed, machine-verifiable record of what was done.

Do **not** invoke for:

- pure Q&A / explanations (no code changes needed);
- interactive debugging where the user is driving each step;
- anything that needs to run destructive commands outside the allowed
  write roots.

## Execution entry point

The skill delegates all real work to the local orchestrator script:

```
python scripts/run_self_supervisor_qoder.py --request "<user request>"
```

Optional flags:

- `--request-file PATH` — read the prompt from a file
- `--dry-run`           — produce the plan and task cards, skip writes
- `--force`             — proceed past non-ok preflight warnings
- `--json`              — also emit the final delivery report to stdout

After the orchestrator finishes, validate the delivery:

```
python scripts/verify_delivery.py --json
```

## Workflow stages

1. **preflight** — environment sanity (git, python, qoder CLI, cleanliness)
2. **plan**      — deterministic heuristic: single-task vs multi-task
3. **write**     — delegates to qoder CLI when a `qoder_exec_template` is
                   configured; otherwise writes task cards under
                   `.qoder/state/tasks/` and scratch notes under
                   `artifacts/` for human completion (advisory mode)
4. **test**      — invokes the single unified test runner
                   `scripts/run_tests.py`
5. **review**    — diff-based (added/modified/removed) with scope check
6. **audit**     — deterministic pass/fail gates (tests, scope, plan shape)
7. **seal**      — writes `artifacts/delivery_report.json`

## Final outputs

- `artifacts/delivery_report.json` — machine-readable, stage-by-stage
  evidence; this is the sealed record of the run.
- `artifacts/user_acceptance.md`   — human-readable acceptance checklist
  generated by `verify_delivery.py`.
- `.qoder/state/plan.json`         — the plan actually used.
- `.qoder/state/tasks/*.md`        — one card per planned task.

## Acceptance contract

`verify_delivery.py` emits one of:

- `ready_for_acceptance` — sealed, tests green, scope respected.
- `ready_with_warnings`  — sealed but there is non-blocking drift
                           (e.g., advisory mode, minor status change).
- `warn_strict`          — warnings present and `--strict` was set.
- `blocked`              — missing report, test regression, or scope
                           violation. Do not accept.

## Configuration

All behavior is controlled by `supervisor_config.json` at the repo root.
See `README.md` for the full field reference.
